# UX — 画面設計 / フロー / デザイン原則

> TL;DR
> UXはこのプロジェクトの最優先事項。どのアプリよりも使いやすくを目指す。
> モバイル向けの快適さをWebで先に実現し、アプリへ自然に引き上げる設計とする。
> カラーパレットは暖色（活力・食欲）×アクセント緑（健康）の組み合わせ。

---

## Decision

### UX 原則（必ず守ること）

1. **0クリック原則** — 毎日使う操作は3タップ以内で完了する
2. **即時フィードバック** — 入力・記録・削除はアニメーションで即反応する
3. **空状態（Empty State）を設計する** — データゼロの画面でも心地よい誘導を出す
4. **エラーを怖くしない** — エラーメッセージは「何をすれば解決するか」を必ず示す
5. **モバイルファースト** — WebでもiOS/Android感覚のジェスチャー・余白・フォントサイズ
6. **美しさが継続を作る** — 機能より先にビジュアルの品質を担保する

> モバイルアプリ版は「どのアプリよりもUXにこだわる」を開発原則に設定する。

## 主要フロー

### 初回利用フロー

1. ランディングページ表示
2. 「Google でログイン」ボタンタップ
3. Google 認証
4. ダッシュボードへリダイレクト（初回：目標カロリー設定へ促す）
5. 目標カロリー設定（設定画面またはオンボーディングモーダル）
6. ダッシュボードへ戻る

### 食事記録フロー（毎日の主フロー）

1. ダッシュボードの「+ 追加」ボタンタップ
2. 食品検索画面 → キーワード入力
3. 検索結果から選択
4. 量を調整（デフォルト 100g or 1個）
5. 「記録する」ボタン → ダッシュボードに即反映（アニメーション）

### アプリ誘導モーダルフロー（Week 2）

1. ログイン後、初回セッションでモーダルを自動表示
2. 「通知を受け取る」→ CTA（App Store 事前登録 or メール収集）
3. 「今はいい」→ モーダルを閉じ、30日間非表示

### 今日の記録がない場合の画面遷移

ログイン後、**今日まだ1件も記録がない**状態でダッシュボード（ホーム）に来たユーザー向けに、記録へ誘導する入口を変える案。負荷をかけすぎず「今日の1件」を促すことを目的とする。

| 案 | 内容 | メリット |
|---|---|---|
| **案1** | 今日の記録が0件のとき、**ダッシュボードではなく記録画面（食品検索など記録の入口）に直接遷移する** | 迷わせずすぐ「記録する」に集中できる |
| **案2** | ホーム（ダッシュボード）を表示し、**モーダルを出して「今日はまだ記録がありません」と案内 → ボタンで記録画面へ遷移** | ホームの存在を伝えつつ、明示的に記録を促せる |

> **Open Question:** 案1・案2のどちらを採用するか（A/B検証の対象にするかは別途検討）。

## トーン & マナー

| 項目 | 方針 |
|---|---|
| 言葉のトーン | フレンドリー・前向き。命令形を避け、誘導型にする |
| アニメーション | Subtle（さりげなく）。過剰なアニメは避ける。ライブラリは Framer Motion を利用する |
| 余白 | ゆとりのある余白（8px グリッド基準） |
| フォント | システムフォント優先（Inter or Noto Sans JP） |
| アイコン | [Heroicons](https://heroicons.com/) を利用する（シンプル・線形） |
| イラスト / 画像 | 生成AIで用意した料理画像（統一感ある明るいスタイル） |

## Rationale

- モバイルファーストで Web を設計することで、将来の React Native 移植コストを下げる
- カラーパレットは「食欲促進（暖色）× 健康意識（緑）」の組み合わせでブランドを明確化
