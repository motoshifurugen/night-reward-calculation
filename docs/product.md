# Product — ビジョン / MVP / ロールアウト

> TL;DR
> 「夜の報酬計算（Night Reward Calculation）」は、食事・カロリー管理をシンプルかつ楽しくするWebアプリ。
> 広告なしでスタートし、2週間後にモバイルアプリへの誘導を開始する。
> まず Web MVP を最速で届け、UX品質でモバイルアプリへ引き上げる。

---

## Decision

### Vision

> ユーザーが「今夜の食事」を迷わず、罪悪感なく選べるようにする。
> シンプルで美しいUIが、継続の動機になる。

### 対象ユーザー（Primary Persona）

| 属性 | 内容 |
|---|---|
| 年齢層 | 20〜35歳 |
| 生活スタイル | 仕事が忙しく、食事管理を習慣化したいが続かない人 |
| 動機 | ダイエット・体型維持、または健康意識 |
| デバイス | スマートフォン（Web）→ 将来：スマートフォンアプリ |
| 技術リテラシー | Google アカウントを普通に使える |

> **Open Question:** セカンダリペルソナ（例：筋トレ勢、医療的食制限あり）を定義するか？

---

## MVP 範囲

### やること（In Scope）

- [ ] Google Sign-In によるログイン / ログアウト
- [ ] 食品検索（seedデータから）
- [ ] 1日のカロリー記録・合計表示
- [ ] 目標カロリー設定
- [ ] 記録履歴（直近7日間）
- [ ] ログイン後「モバイルアプリ誘導モーダル」（2週間後リリース）

### やらないこと（Out of Scope for MVP）

- 栄養素詳細（たんぱく質、脂質など）の詳細表示
- ソーシャル機能（シェア、フォロー）
- 広告掲載
- プッシュ通知
- オフライン対応
- 料金プラン / サブスクリプション
- バーコードスキャン

> **Open Question:** 目標カロリーのデフォルト値は何 kcal にするか？（例：2000 kcal）
> **Open Question:** 記録履歴は7日間で十分か、30日間対応するか？

---

## Rationale

- 広告なしでスタートすることで、UX品質に集中できる
- seedデータ（JSON）でカロリーDBをコントロール下に置き、初期の依存を減らす
- 料理画像は生成AIで用意し、著作権リスクをゼロにする

---

## ロールアウト計画

```
Week 0     : MVP 開発開始
Week 1–2   : Web MVP リリース（ログイン・記録・履歴）
Week 2+    : ログイン後モーダル「モバイルアプリがもうすぐ来る」を追加
Week 4–6   : モバイル版 React Native 開発開始（予定）
```

### モバイル誘導モーダル仕様（Week 2 追加予定）

- トリガー：ログイン後の最初のセッション（1回のみ表示、または1日1回）
- 内容：「より快適なモバイルアプリが近日公開」
- CTA：「通知を受け取る」（メールアドレス or App Store 事前登録）
- 却下：「今はいい」→ 30日間非表示

> **Open Question:** モーダルの表示頻度は「ログインごとに1回」か「1週間に1回」か？
> **Open Question:** CTA は App Store 事前登録リンクか、メール収集フォームか？
